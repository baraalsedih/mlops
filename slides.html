Choosing the right tool
W&B Artifacts — metadata only
  • Small datasets (< a few GB)
  • Non-sensitive data
  • Log schema, hash, split sizes — not raw rows
  • Lineage visible in W&B UI automatically
artifact = wandb.Artifact(
  "wine-dataset-meta", type="dataset",
  metadata={"hash": hash_train,

# Only add a small JSON — not the CSV

DVC — large / sensitive data
  • Large datasets (GBs / TBs)
  • Data stays on your servers
  • Git stores only a pointer file (MD5 + size)
  • DVC fetches bytes from your configured remote

Key takeaways
W&B & versioning
Every experiment is a run. Add wandb.init + wandb.log and your training script becomes reproducible and comparable by default.
Don't upload raw data. Log metadata — hash, schema, split sizes — for audit trails. Use DVC for the actual bytes when they're large or client-sensitive.
Aliases over version numbers. Deployment code references :production, not :v7. Promotion and rollback become one-click operations.
git + DVC + W&B = complete picture. Code reproducibility, data reproducibility, and experiment visibility are separate concerns solved by separate tools.

Context
ML makes CI/CD harder
Traditional software CI/CD
  • Pass/fail is deterministic
  • Output = lines of code (text)
  • Test coverage is straightforward
  • Rollback = revert a commit
ML systems — extra challenges
  • Pass/fail depends on data + code + randomness
  • Output = a binary model file
  • Training can take hours or days
  • Rollback = previous model version
  • New failure mode: silent model degradation
We need the same CI/CD habits, plus a model quality gate — a test that blocks deployment if accuracy or latency misses the threshold.
00:47
Docker
Why containerize ML code?
Reproducibility

"Works on my machine" disappears. The container carries its own Python, CUDA, and library versions everywhere.

Portability

Same image runs on a laptop, CI server, cloud VM, or Kubernetes pod without modification.

Isolation

Different models with conflicting dependencies co-exist safely. Training and serving can use different base images.

Multi-stage builds — keep images lean: a heavy builder stage installs dependencies, then only the installed packages are copied into a minimal runtime image. Final image is ~180 MB instead of ~800 MB.

Live demo — 02_cicd/Dockerfile + docker-compose.yml
Dockerfile + Docker Compose
# ── Stage 1: builder ─────────────
FROM python:3.11-slim AS builder

WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir \
    --prefix=/install \
    -r requirements.txt

# ── Stage 2: runtime ─────────────
FROM python:3.11-slim AS runtime

WORKDIR /app
# Copy only installed packages
COPY --from=builder /install /usr/local
COPY src/ ./src/

ENTRYPOINT ["python", "src/train.py"]
docker-compose.yml
services:
  train:
    build: .           # uses our Dockerfile
    volumes:
      - ./models:/app/models
    environment:
      - PYTHONUNBUFFERED=1
    command: ["python", "src/train.py"]
Build & run with Compose
# Build image + start in one command
docker compose up --build

# Or separately:
docker compose build
docker compose up

# Stop and remove containers
docker compose down
Why Compose over raw docker run?
  • Volumes, env vars & services declared once in a file
  • Easy to add a serving container later (depends_on: train)
  • Same command works for every team member

Live demo — test_model_quality.py
The quality gate in code
ACCURACY_THRESHOLD = 0.85
LATENCY_MS = 100

def test_model_accuracy(trained_model, test_set):
    X_test, y_test = test_set
    preds = trained_model.predict(X_test)
    acc = accuracy_score(y_test, preds)
    assert acc >= ACCURACY_THRESHOLD, (
        f"Accuracy {acc:.4f} below threshold.
         Deployment blocked."
    )

def test_inference_latency(trained_model, test_set):
    sample = test_set[0][[0]]
    start = time.perf_counter()
    trained_model.predict(sample)
    ms = (time.perf_counter() - start) * 1000
    assert ms < LATENCY_MS, (
        f"Latency {ms:.1f}ms — too slow."
    )
In ml-pipeline.yml
- name: Run model quality gate
  working-directory: 02_cicd
  run: pytest tests/test_model_quality.py -v
  # Pipeline STOPS if tests fail
Fail fast. A model that misses the threshold is caught in CI before a Docker image is built or a deploy is triggered. Saves compute and prevents silent regressions reaching staging.

Key takeaways
CI/CD for ML
Container = reproducibility. Multi-stage Dockerfiles keep images lean while ensuring the training environment is identical across dev, CI, and prod.
Four-job pipeline. test → train+validate → build → deploy. Each job only runs if the previous passes. The quality gate is the key addition over standard software CI.
Three test layers. Data schema, model quality, and API smoke tests cover different failure modes. Layer 2 is the gate that actually blocks bad models from deploying.
Makefile as local shortcut. make train, make test, make docker-build — same commands developers run locally and CI runs remotely.

Root causes
Why models degrade in production
Data drift

The input distribution changes. Trained on winter data; summer arrives. Features that were stable begin shifting. The model's assumptions are no longer valid.

Concept drift

The relationship between inputs and outputs changes. Fraud patterns evolve; customer behavior shifts. The model is now confidently wrong.

Model degradation

Performance gradually erodes due to accumulated drift. Often not a single event — it's a slow leak invisible without monitoring. The most dangerous failure mode.

Pipeline / data issues

Missing features at serving time, upstream schema changes, ETL bugs. These cause silent mispredictions that look like drift but are actually engineering failures.

01:40
Drift taxonomy
Types of drift
Covariate shift
P(X) changes · P(Y|X) stable

Input distribution shifts but the underlying pattern holds. Model still works, just less confidently.

Most common in practice
Prior probability shift
P(Y) changes · P(X|Y) stable

Label prevalence shifts — fraud rate spikes, but fraud features remain similar. Impacts calibration.

Common in imbalanced domains
Concept drift
P(Y|X) changes

The fundamental pattern changes — new fraud tactics, new user segments. Model is confidently wrong.

Hardest to detect
Sudden

Abrupt change — new data source, product change. Sharp metric drop.

Gradual

Slow erosion over weeks/months. Easy to miss. Most damaging.

Recurring

Seasonal — needs periodic scheduled retraining, not just on degradation.

01:47
Detection
Statistical tests for drift
KS Test
Kolmogorov-Smirnov

For continuous features. Compares CDFs of reference vs production distribution. Returns a p-value — drift detected if p < 0.05. Non-parametric, no assumptions.

numericalnon-parametric
PSI
Population Stability Index

For continuous features (binned). Measures magnitude of shift. PSI < 0.1: stable · 0.1–0.2: monitor · > 0.2: significant drift, act now.

numericalpopular in finance/risk
Chi-squared
Pearson's χ²

For categorical features. Tests if observed category frequencies differ from expected. Ideal for monitoring label distribution shift and categorical feature drift.

categoricallabel monitoring

Tooling
Evidently AI

Open-source ML monitoring library. Generates HTML reports and metric snapshots for data quality, data drift, and model performance.

Core concepts
  • Report — static HTML with visualizations for the team
  • Test Suite — pass/fail checks with configurable thresholds
  • Presets — curated bundles: DataDriftPreset, DataQualityPreset, RegressionPreset…
Evidently applies the KS, PSI, and chi-squared tests internally and surfaces results as human-readable HTML reports — no manual stat test coding required.
from evidently import Report
from evidently.metric_preset import (
    DataDriftPreset,
    DataQualityPreset,
)

report = Report(metrics=[
    DataDriftPreset(),
    DataQualityPreset(),
])

report.run(
    reference_data=reference_df,
    current_data=production_df,
)

# HTML report for the team
report.save_html("monitoring.html")

# Programmatic access for alerts
result = report.as_dict()
drifted = result["metrics"][0][
    "result"]["share_of_drifted_columns"]

Key takeaways
Production monitoring
Models decay silently. Without monitoring, degradation is invisible until a business metric collapses. Set up drift detection before you deploy, not after.
Know your drift type. Covariate shift (inputs change) vs concept drift (relationship changes) require different responses. Monitor both input distributions and prediction distributions.
Statistical tests are your first alarm. KS for continuous features, chi-squared for categorical, PSI as a business-friendly score. Set thresholds and automate the check per batch.
Evidently makes it concrete. HTML reports your team can read, test suites that integrate into pipelines, historical trend dashboards.
Pre-prod prevents prod fires. Shadow mode + canary deployment catches regressions before they impact live users and business KPIs.
